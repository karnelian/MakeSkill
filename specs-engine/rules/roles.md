# Roles — 역할 시스템 v5.4.1 (Full Industry Edition + 3-Tier)

---

## 핵심 원칙

- 역할 = **고려해야 할 관점** (실제 채용 인원이 아님)
- 규모와 무관하게 전문 관점을 최대한 배치 — **"회사를 운영하는" 수준의 전문성**
- 프로그래머/개발자는 기획 토론에서 **구현 가능성 검증** 역할만
- 1인 모드에서는 AI가 전 역할 수행
- v5.4: 유형별 역할을 **업계 실제 직군명으로 전수 확장** (게임 118, 웹 82, 모바일 21, API 12, 데스크톱 13, 확장 7, 크로스커팅 45)
- v5.4.1: **3-Tier 역할 소환 시스템** 도입 — 전체 조직도는 유지하되 실제 토론은 효율적으로

---

## 3-Tier 역할 소환 시스템 (v5.4.1)

> 300+명 전체가 매번 토론하면 컨텍스트 낭비. Tier 분류로 **필요한 관점만 적시 소환**.

| Tier | 이름 | 규모 | 소환 조건 | runner.js 동작 |
|------|------|------|----------|---------------|
| **T1** | 핵심팀 | 유형당 5~8명 | **항상** 프롬프트 주입 | `ROLE_TIERS[type].t1` |
| **T2** | 전문가 | 유형당 15~25명 | 섹션 키워드 매칭 시 **자동 소환** | `matchT2Experts()` |
| **T3** | 레퍼런스 | 나머지 전부 | 딥다이브 요청 시 **수동 참조** | roles.md 전체 조직도 |

### T1 핵심팀 (유형별)

| 유형 | T1 핵심팀 |
|------|----------|
| 게임 | 게임 디렉터, 시스템 기획자, 아트 디렉터, 테크니컬 디렉터, 프로듀서, BM 기획자, QA 리드, 게임 분석가 |
| 웹 SaaS | 프로덕트 디자이너, 프론트엔드, 백엔드, DevOps/SRE, 보안, PM, QA, 프로덕트 분석가 |
| 모바일 | 모바일 UX, iOS, Android, 백엔드 설계자, QA, 분석 |
| API | API 아키텍트, API 엔지니어, DX, 신뢰성, 보안 |
| 데스크톱 | 데스크톱 UX, 크로스플랫폼, OS 통합, 보안, 업데이트 |
| 크롬 확장 | 프론트엔드, 서비스 워커, 보안/프라이버시, 웹스토어 |

### T2 자동 소환 예시

```
섹션: "전투 시스템과 밸런스 설계"
  → 키워드 매칭: "전투", "밸런스"
  → 자동 소환: 전투 기획자, 밸런스 기획자

섹션: "인증 및 소셜 로그인 설계"
  → 키워드 매칭: "인증", "소셜"
  → 자동 소환: 인증·인가 엔지니어, 소셜 기획자
```

### T3 참조 방법

토론 중 "이 부분은 더 전문적인 의견이 필요하다" → roles.md의 해당 카테고리에서 전문가 직접 소환

## 공통 기획 역할 (5명, 모든 유형)

| 역할 | 관점 | 우선시하는 것 |
|------|------|-------------|
| **CEO** | 비전/전략 | 사업 방향, 최종 판정 (A급 결정) |
| **마케터** | 시장/채널/성장 | 고객 획득, 포지셔닝, GTM |
| **UX 리서처** | 사용자/제품 | 사용자 니즈, 행동 패턴 |
| **CTO** | 기술/실현 가능성 | 아키텍처, 기술 리스크, 공수 |
| **CFO** | 재무/수익/지표 | 수익성, 비용 구조, ROI |

### 검토 의무

| 역할 | 확인 사항 |
|------|----------|
| CEO | "비전과 일치하는가?" |
| 마케터 | "고객에게 전달 가능한가?" |
| UX 리서처 | "사용자가 정말 원하는가?" |
| CTO | "기술적으로 가능한가? 공수는?" |
| CFO | "비용 대비 효과가 있는가?" |

### 토론: [결정할 주제]

## 역할 참여 매트릭스 (v5.4)

> 모든 역할이 모든 Phase에 참여하지 않는다.
> Phase별 필요한 관점만 소환하여 컨텍스트 효율을 유지한다.

| Phase | 참여 역할 기준 |
|-------|--------------|
| **Research** | 공통 5명 (CEO/마케터/UX리서처/CTO/CFO) |
| **Discovery** | 공통 5명 + 유형별 핵심 3~5명 |
| **Planning** | 공통 5명 + 유형별 전원 중 `Planning` 태그 역할 + 크로스커팅 필요 역할 |
| **Design** | CTO + 유형별 전원 중 `Design` 태그 역할 + 크로스커팅 필요 역할 |
| **Verify** | CFO + CTO + 유형별 `Verify` 태그 역할 |
| **Implementation** | CTO + 유형별 `Impl` 태그 역할 |
| **Presentation** | CEO + 마케터 + UX리서처 |
| **Build→Deploy** | CTO + DevOps + 유형별 인프라 역할 (게이트 이후) |

### 역할 소환 원칙

1. **Phase 태그 기반**: 각 역할의 `참여 Phase` 컬럼에 명시된 Phase에서만 소환
2. **크로스커팅 역할**: 프로젝트 필요에 따라 어떤 유형이든 추가 소환 가능
3. **규모 적응**: 소규모 프로젝트에서는 핵심 역할만, 대규모에서는 전체 소환
4. **1인 모드**: AI가 모든 역할의 관점을 순회하며 검토

## 참고

- 역할은 회사 규모에 따라 한 사람이 여러 역할을 겸하거나, 하나의 역할이 더 세분화될 수 있습니다.
- 스타트업: 풀스택 1명이 프론트+백엔드+인프라 전부 → 그래도 **관점은** 각 역할별로 검토
- 대기업: "iOS 성능 엔지니어"처럼 극도로 전문화된 역할이 독립 존재
- AI/ML 역할은 2024-2026년 급격히 증가 추세 (AI Engineer, Prompt Engineer, AI Safety 등)
- Korean 용어는 한국 IT/게임 업계 실제 통용 표현 기준 (영문 그대로 쓰는 경우 포함)
