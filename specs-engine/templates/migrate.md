# Migrate — 스킬 버전 마이그레이션

스킬/템플릿 버전이 업그레이드되면, **기존 프로젝트의 산출물**이 새 규칙을 충족하지 못할 수 있다.
이 템플릿은 기존 프로젝트를 새 버전에 맞게 **일괄 업데이트**한다.

---

## 컨텍스트 엔지니어링

읽어야 할 파일:
- `specs/` 하위 전체 프로젝트 목록
- 각 프로젝트의 `STATE.md`
- 마이그레이션 대상 파일 (변경 규칙별)

---

## 프로세스

### Step 1: 프로젝트 스캔

각 프로젝트의 STATE.md를 읽어 완료된 Phase 확인.

### Step 2: 마이그레이션 필요성 판단

각 프로젝트 × 각 규칙에 대해:
- 해당 Phase가 완료됐으면 → 마이그레이션 대상
- 해당 Phase가 미완료면 → 스킵 (새 버전이 처음부터 적용)

```markdown
## 마이그레이션 스캔 결과

| 프로젝트 | 변경 규칙 | 대상 파일 | 상태 |
|----------|----------|----------|------|
| {name} | {rule} | {files} | ⬜ 필요 / ✅ 이미 적용 |
```

### Step 3: 일괄 마이그레이션

영향도 낮은 것부터 → 높은 것 순서:
1. 검증 섹션 추가 (research/, context.md)
2. 태그/라벨 추가 (impl/tasks/)
3. 수치 교차검증 보강 (planning/, design/)
4. 스코프 정합성 수정 (presentation/, infographic/)
5. SSOT 앵커 수치 대조 (context.md ↔ ch07)

### Step 4: 검증

마이그레이션 완료 후 verify Phase 실행하여 전체 검증.

### Step 5: 리포트

```markdown
## 마이그레이션 완료 리포트

| 프로젝트 | 수정 파일 | 수정 항목 | 검증 |
|----------|----------|----------|------|
| {name} | {N}개 | {summary} | ✅ |
```

## 실행 모드

| 모드 | 명령 | 설명 |
|------|------|------|
| 전체 | `--all` | specs/ 하위 전체 프로젝트 |
| 단일 | `{프로젝트명}` | 특정 프로젝트만 |
| 특정 규칙 | `{프로젝트명} --rule {규칙}` | 특정 규칙만 적용 |
| 미리보기 | `--dry-run` | 실제 수정 없이 리포트만 |
