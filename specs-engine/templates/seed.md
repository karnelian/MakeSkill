Deep Questioning으로 핵심 의도를 파악하고, 유형 판별 및 역할 배정을 수행한다.

## 프로세스

### 1. 아이디어 수집

사용자가 제공한 아이디어 원문을 그대로 기록한다.
인자가 있으면 그것이 아이디어. 없으면 사용자에게 질문한다.

### 2. Deep Questioning

사용자에게 핵심 인텐트를 추출하는 질문을 한다. **한 번에 모아서** 질문한다.

필수 질문:
1. "왜 이걸 만들고 싶으세요?" (동기/비전)
2. "누구를 위한 건가요?" (타겟)
3. "성공하면 어떤 모습인가요?" (목표/KPI)

선택 질문 (필요시):
- "이미 시도해보거나 참고한 것이 있나요?" (맥락)
- "예산이나 기간 제약이 있나요?" (제약)

규칙: 여러 번 나눠 묻지 않는다. 한 번에 최대 3~5개 질문.

### 3. 프로젝트 유형 자동 판별

6개 유형 중 키워드 매칭으로 자동 판별:
- 게임 / 웹 SaaS / 모바일 앱 / API 서비스 / 크롬 확장 / 데스크톱 앱

복합 유형 시 사용자에게 확인한다.

### 4. 역할 자동 배정

- 공통 5명: CEO, 마케터, UX 리서처, CTO, CFO
- 유형별 전문가:
  - 게임: 8명 (게임 디자이너, 레벨 디자이너, 내러티브, 아트 디렉터, 사운드, 서버, 과금, QA)
  - 웹 SaaS: 6명 (프론트, 백엔드, 그로스, 데이터, DevOps, QA)
  - 모바일: 6명 (모바일 개발, 백엔드, 그로스, ASO, 성능, QA)
  - API: 6명 (API 설계, 백엔드, 문서화, DevOps, 보안, QA)
  - 크롬 확장: 5명 (확장 개발, 백엔드, 프라이버시, ASO, QA)
  - 데스크톱: 5명 (데스크톱 개발, 시스템, 크로스플랫폼, 배포, QA)

### 5. seed.md 생성

```markdown
# Seed

## 아이디어
(사용자 원문 그대로)

## 인텐트
- 동기: (왜)
- 타겟: (누구)
- 성공 기준: (어떤 모습)
- 맥락: (참고/시도한 것)
- 제약: (예산/기간)

## 판별 결과
- 유형: {유형}
- 배정된 역할:
  - 공통: CEO, 마케터, UX 리서처, CTO, CFO
  - 전문가: (유형별 목록)
```

## SSOT 규칙

- 성공 기준의 정량화 유도: 가능하면 숫자로 기록 (예: "Y1 MRR $10K"). 이 수치가 이후 context.md 핵심 수치의 앵커가 된다.
- 제약 조건의 명시적 기록: 예산/기간 제약은 seed.md에 명확히 기록. 이후 인프라 비용의 상한선이 된다.
- STATE.md에 15 Phase 전체 포함 (Verify 3개 + Infographic + Build/Review/QA/Deploy 누락 금지).
