## Ch04: 사용자 분석

**역할**: UX 리서처 주도 / 마케터, CEO 검토

### 필수 내용

1. **타겟 사용자 정의** --- 1차/2차 타겟, 제외 대상
2. **인구통계 프로필** --- 연령, 지역, 직업, 소득 (표)
3. **행동 패턴** --- 사용 맥락, 빈도, 디바이스
4. **페인 포인트** --- 현재 겪는 문제점 (우선순위 표)
5. **니즈 & 기대** --- Jobs to be Done 프레임워크
6. **페르소나** --- 2~3개 상세 프로필
7. **사용자 여정 맵** --- 인지 → 탐색 → 사용 → 유지

### 분량 목표

- 최소: 120줄 / 목표: 200줄

## Ch05: 제품 정의 (최대 섹션)

**역할**: UX 리서처 주도 (+ CTO 기술 검증) / CTO, CEO 검토

### 필수 내용

1. **제품 비전** --- 한 문장 + 엘리베이터 피치
2. **핵심 가치 제안** --- Value Proposition Canvas
3. **기능 목록** --- Feature ID 체계(F01~) 사용

| 우선순위 | ID | 기능 | 설명 | 페르소나 | MVP |
|---------|-----|------|------|---------|-----|
| P0 | F01 | {필수 기능} | ... | 페르소나1 | v1 |
| P1 | F02 | {핵심 기능} | ... | 페르소나1,2 | v1 |
| P2 | F03 | {확장 기능} | ... | 페르소나2 | v2 |

4. **기능 상세 설명** --- P0, P1 기능별:
   - 기능명 + Feature ID
   - 사용자 스토리: "~으로서, ~를 하고 싶다, ~이기 때문에"
   - 수용 기준 (Acceptance Criteria)
   - 엣지 케이스
   - 기술적 고려사항 (CTO 코멘트)

5. **화면/인터페이스 설계** --- 주요 화면 목록 (표), 화면 전환 플로우, 핵심 화면별 요소 설명
6. **사용자 시나리오** --- 주요 플로우 3~5개
7. **MVP 범위** --- 포함(v1) / 미포함(v2) 표
8. **프리미엄 비교표** --- Free/Pro 기능 비교 (이 표가 _contract.md C3의 정본)
9. **비기능 요구사항** --- 성능, 보안, 접근성 기준

### Feature ID 체계

- F01~ 순차 부여. context.md + seed.md 인텐트에서 도출 가능한 범위만
- P0/P1은 인텐트에서 필연적으로 도출되는 기능만
- P2/P3 확장 기능은 최대 4개로 제한
- v1/v2 경계를 명확히 구분하여 모든 downstream이 혼동 없이 참조

### 분량 목표

- 최소: 250줄 / 목표: 400줄

## _contract.md 생성 (Ch05 완료 직후 필수)

Ch05 작성 완료 직후, `specs/{프로젝트명}/planning/_contract.md`를 생성한다.
이후 Ch06~Ch12+Ch01은 context.md를 직접 해석하지 않고 contract만 참조한다.

### Contract 구조 (7 섹션)

#### C1. 가격 정책
- **출처**: context.md Locked Decisions
- **추출**: 플랜별 가격/포함 내용 표
- **검증**: 후속 챕터의 가격이 C1과 다르면 FAIL

#### C2. 핵심 수치 SSOT
- **출처**: context.md 핵심 수치 SSOT 테이블
- **추출**: 전체 테이블 그대로 복사 (지표, 값, 시점, 비고)
- **검증**: Ch01/Ch02/Ch07/Ch09에서 동일 지표 사용 시 C2 값과 일치해야 함

#### C3. 프리미엄 비교표
- **출처**: Ch05 프리미엄 비교표 (정본)
- **추출**: 셀 단위 그대로 복사
- **검증**: Ch07/Ch01에서 비교표 사용 시 C3과 셀 단위 1:1 일치. 항목 추가/변경/삭제 → FAIL

#### C4. Feature Registry (F01~)
- **출처**: Ch05 기능 목록 + MVP 범위표
- **추출**: Feature ID, 기능명, 우선순위, v1 플래그, 마일스톤 배정

| ID | 기능명 | 우선순위 | v1 | 마일스톤 | 비고 |
|----|--------|---------|-----|---------|------|
| F01 | ... | P0 | v1 | M2 | |
| F02 | ... | P2 | v2 | v2 | v2 기능 |

- **검증**: v2 기능을 M1~M4 타임라인/과금표/기능 매트릭스에 v1으로 사용하면 FAIL

#### C5. 마일스톤 정의
- **출처**: skeleton.md
- **추출**: 마일스톤 이름, 기간, 목표

| 마일스톤 | 이름 | 기간 | 목표 |
|---------|------|------|------|
| M1 | ... | X주 | ... |

- **검증**: 전 챕터에서 마일스톤 언급 시 이 이름을 그대로 사용

#### C6. 기술 스택
- **출처**: context.md Software Stack (Locked)
- **추출**: 카테고리, 선택, 결정 시점 그대로 복사
- **검증**: Ch06에서 스택명/버전이 C6과 다르면 FAIL

#### C7. 인프라 비용 (Ch06 작성 후 갱신)
- **출처**: Ch06 인프라 비용 테이블 (정본)
- **추출**: Ch06 완료 후 "인프라 월비용: ~${총합}" 기록
- **검증**: Ch07/Ch01/Ch12는 C7 총합만 인용. 개별 항목 분해 금지. 역방향 변경 금지

### 생성 규칙

1. **생성 시점**: Ch05 작성 완료 직후 (C1~C6). C7은 Ch06 완료 후 갱신
2. **갱신 조건**: Ch05가 이터레이션(v2/v3)되면 contract 재생성 필수
3. **버전 기록**: contract 하단에 `생성 기준: ch05 v{N}, {날짜}` 기록
4. **수동 편집 금지**: 자동 생성/갱신만 허용
